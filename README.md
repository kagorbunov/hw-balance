# hw-balance

Сервис для вывода средств с защитой от дублей и двойного списания.

## Стек

- Go 1.22+
- PostgreSQL 14+
- REST API

## Структура проекта

```
internal/
├── controller/http  — HTTP-хендлеры, auth, маппинг ошибок
├── adapter/postgres — работа с БД
├── usecase          — бизнес-логика
├── dto              — request/response структуры
└── model            — доменные модели и ошибки
```

## Жизненный цикл withdrawal

```
[create] → pending → [confirm] → confirmed
                  ↘ [cancel] → cancelled (средства возвращаются)
```

- `pending` — средства списаны, ожидает подтверждения
- `confirmed` — вывод завершён, терминальный статус
- `cancelled` — вывод отменён, средства возвращены на баланс

## Как работает защита от двойного списания

Вся логика создания withdrawal выполняется в одной транзакции:

1. Вставляем `idempotency_key` через `INSERT ... ON CONFLICT DO NOTHING`
2. Если ключ уже был — проверяем payload hash:
   - совпадает → возвращаем существующий withdrawal
   - не совпадает → 422
3. Блокируем строку баланса через `SELECT ... FOR UPDATE`
4. Проверяем что хватает денег
5. Списываем и создаём withdrawal
6. Записываем операцию в `balance_ledger` для аудита
7. Связываем idempotency_key с withdrawal_id

`FOR UPDATE` гарантирует что параллельные запросы на один баланс выполняются последовательно.

Если два запроса с одним idempotency_key приходят одновременно — второй ждёт завершения первого (retry с интервалом 50ms, до 5 попыток).

## Аудит операций (Ledger)

Все изменения баланса записываются в `balance_ledger`:

| operation_type | amount | описание |
|---------------|--------|----------|
| withdrawal_create | отрицательный | списание при создании |
| withdrawal_cancel | положительный | возврат при отмене |

Ledger позволяет:
- Восстановить историю изменений баланса
- Провести аудит операций
- Отладить расхождения баланса

## Запуск

```bash
docker compose up --build
```

API будет доступен на `http://localhost:18088`

Токен для авторизации: `secret-token`

Тестовый баланс: user_id=1, 1000000 USDT

### Без Docker

```bash
# Поднять postgres и применить миграции из db/migrations/

export DATABASE_URL='postgres://app:app@localhost:5432/hw_balance?sslmode=disable'
export AUTH_TOKEN='secret-token'
go run ./cmd/server
```

## API

### POST /v1/withdrawals

Создание вывода.

```bash
curl -X POST http://localhost:18088/v1/withdrawals \
  -H "Authorization: Bearer secret-token" \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": 1,
    "amount": 100,
    "currency": "USDT",
    "destination": "TRC20-wallet-address",
    "idempotency_key": "order-123"
  }'
```

Ответы:
- 201 — создано (или идемпотентный повтор)
- 400 — невалидные данные
- 401 — нет токена или неверный
- 409 — недостаточно средств
- 422 — idempotency_key уже использован с другими данными

### GET /v1/withdrawals/{id}

```bash
curl http://localhost:18088/v1/withdrawals/uuid-here \
  -H "Authorization: Bearer secret-token"
```

Ответы:
- 200 — ok
- 400 — невалидный uuid
- 404 — не найден

### POST /v1/withdrawals/{id}/confirm

Подтверждение вывода. Переводит статус из `pending` в `confirmed`.

```bash
curl -X POST http://localhost:18088/v1/withdrawals/uuid-here/confirm \
  -H "Authorization: Bearer secret-token"
```

Ответы:
- 200 — подтверждено
- 400 — невалидный uuid
- 404 — не найден
- 409 — не в статусе pending

### POST /v1/withdrawals/{id}/cancel

Отмена вывода. Переводит статус из `pending` в `cancelled` и возвращает средства на баланс.

```bash
curl -X POST http://localhost:18088/v1/withdrawals/uuid-here/cancel \
  -H "Authorization: Bearer secret-token"
```

Ответы:
- 200 — отменено, средства возвращены
- 400 — невалидный uuid
- 404 — не найден
- 409 — не в статусе pending

## Trade-offs и архитектурные решения

### Почему FOR UPDATE вместо advisory locks?

**Выбрано:** `SELECT ... FOR UPDATE` на строке баланса.

**Плюсы:**
- Простота: блокировка привязана к транзакции, автоматически освобождается
- Атомарность: блокировка и чтение в одном запросе
- Не требует дополнительных таблиц или координации

**Минусы:**
- Блокирует всю строку баланса, параллельные операции сериализуются
- При высокой нагрузке на один баланс может стать bottleneck

**Альтернатива — advisory locks:**
- Более гибкие, можно блокировать по произвольному ключу
- Требуют явного управления (lock/unlock)
- Сложнее отлаживать deadlocks

**Вывод:** Для текущих требований `FOR UPDATE` достаточно. При масштабировании можно рассмотреть шардирование балансов или очередь операций.

### Почему отдельная таблица idempotency_keys?

**Выбрано:** Отдельная таблица с `withdrawal_id = NULL` пока операция не завершена.

**Плюсы:**
- Обрабатывает race condition: второй запрос видит что ключ занят, но withdrawal ещё нет
- Retry-механизм ждёт завершения первого запроса
- Чёткое разделение: идемпотентность vs бизнес-сущность

**Альтернатива — UNIQUE constraint только в withdrawals:**
- Проще схема
- Но не обрабатывает случай когда два идентичных запроса приходят одновременно
- Один из них получит constraint violation вместо корректного повтора

### Почему balance_ledger вместо только UPDATE balances?

**Выбрано:** Каждое изменение баланса записывается в `balance_ledger`.

**Плюсы:**
- Полная история: можно восстановить состояние на любой момент
- Аудит: видно кто, когда и почему изменил баланс
- Отладка: легко найти причину расхождений

**Минусы:**
- Дополнительная запись на каждую операцию
- Таблица растёт со временем

**Вывод:** Для финансовых операций аудитность критична. Оверхед на запись минимален.

### Что можно улучшить при масштабировании

1. **Очередь операций**: Kafka/RabbitMQ для асинхронной обработки
2. **Шардирование**: Разнести балансы по нескольким БД
3. **Кэширование**: Redis для hot balances с write-through
4. **Rate limiting**: Защита от DDoS на уровне пользователя
5. **Distributed tracing**: OpenTelemetry для отладки в микросервисах
